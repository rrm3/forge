# Forge v2 - Onboarding, Voice, and Daily Interactions

**Date:** 2026-03-20
**Status:** Draft
**Builds on:** [2026-03-12-ai-tuesdays-design.md](../2026-03-12-ai-tuesdays-design.md)

## Overview

This design evolves Forge from a general chat assistant into a structured AI Tuesdays companion with purpose-driven session types, voice interaction via OpenAI Realtime API, and search powered by the shared ds-internal-mcp LanceDB pipeline. Text chat migrates from SSE to WebSockets via API Gateway WebSocket APIs, decoupling connection management from compute. Voice uses a client-direct ephemeral token pattern that keeps audio off AWS infrastructure entirely, with system prompts injected server-side so they're never exposed to the browser.

## What changes

| Area | Current (v1) | New (v2) |
|------|-------------|----------|
| **First-run experience** | Lightweight chat-based onboarding (greet, validate profile, ask experience level) | Deep conversational intake (placeholder - separate design) |
| **Home screen buttons** | 4 generic suggestions ("Get started with AI", etc.) | 4 purpose-driven actions that create tagged sessions |
| **Session model** | Untyped chat sessions | Tagged sessions with a `type` field |
| **Text chat streaming** | SSE over Lambda Function URL | WebSocket via API Gateway WebSocket API + Lambda |
| **Voice** | None | OpenAI Realtime API (GPT-4o) via client-direct ephemeral token pattern |
| **Search/RAG** | Curriculum-specific LanceDB table (not yet indexed) | Shared LanceDB tables from ds-internal-mcp (Gong, Dovetail, Roadmap, Klue) + department resources |
| **Department resources** | 26 hand-authored curriculum files in repo | Admin-managed markdown files on S3, per-function + global |
| **Existing curriculum** | Wired into search pipeline | Remains in repo but disconnected; not indexed, not searched |
| **Skills/prompts** | 4 skills (onboarding, Tuesday check-in, wrap-up, journaling) | Replaced with session-type prompts (intake, share a tip, I'm stuck, brainstorm, end-of-day) |

## What stays the same

* Lambda compute (now invoked by API Gateway instead of Function URL)
* CloudFront for frontend SPA
* OIDC auth via ds-identity
* Org chart enrichment from SQLite
* DynamoDB + S3 storage
* ReAct agent loop with tools
* React + Vite + Tailwind frontend
* Session management (create, rename, delete, list)

## 1. Session Types and Tagging

Sessions get a `type` field that identifies their purpose. This enables querying across all users ("show me all brainstorms from the marketing team") and drives the initial prompt/skill selection.

### Session types

| Type | Trigger | Purpose |
|------|---------|---------|
| `intake` | First sign-in (auto) | Deep proficiency assessment and role capture |
| `tip` | "Share a Tip or Trick" button | Capture something the user learned |
| `stuck` | "I'm Stuck" button | Brainstorm next steps, draw from curriculum and resources |
| `brainstorm` | "Brainstorm an Opportunity" button | Structured ideation for improving something at DS |
| `wrapup` | "End-of-Day Wrap-up" button | Reflect on the day, capture progress |
| `chat` | Free-form chat (new session without a type button) | General AI conversation |

### Data model change

Add `type` field to the session metadata in DynamoDB:

```json
{
  "user_id": "oidc-sub-uuid",
  "session_id": "uuid",
  "type": "tip",
  "title": "Using Claude for code review",
  "created_at": "2026-03-20T10:30:00Z",
  "updated_at": "2026-03-20T10:35:00Z",
  "message_count": 6
}
```

The `type` field is set at session creation and is immutable. Default is `chat` for sessions created without an action button.

### Querying

A new DynamoDB GSI on `type` enables cross-user queries:
* `type-created_at-index` (GSI): PK=type, SK=created_at

This supports admin queries like "all tips from the last week" or "all intakes in a time range." Note: cross-department queries (e.g., "all intakes from marketing") require joining with profile data and are deferred to the admin dashboard phase.

## 2. Action Buttons

The four suggestion buttons on the empty chat screen are replaced with purpose-driven actions. Each button creates a new session with the corresponding `type` and kicks off the conversation with a type-specific system prompt.

### Button layout

When a user has completed intake and has no active session selected:

```
[Context-aware greeting - see Section 8]

  [ Share a Tip or Trick ]    [ I'm Stuck ]
  [ Brainstorm an Opportunity ]    [ End-of-Day Wrap-up ]
```

The greeting varies based on day, session count, and last activity (e.g., "Ready for AI Tuesday, [Name]?" on Tuesdays, "Week [N] of 12. You're building momentum." after 5+ sessions). See Section 8 for full spec.

Plus a standard "New Chat" button in the sidebar for free-form sessions.

### How buttons work

1. User clicks a button
2. Frontend sends `{"action": "start_session", "type": "tip"}` via the WebSocket connection
3. Backend creates a tagged session, loads the corresponding prompt, and initiates the conversation in one atomic operation
4. The session appears in the sidebar with an appropriate auto-generated title

The `start_session` action accepts an optional `mode: "text" | "voice"` field. Default is `"text"`. If `"voice"`, the backend also creates an OpenAI Realtime session and returns the ephemeral token alongside the first message. Users can toggle between text and voice mid-session at any point.

### Session-type prompts

Each session type has a markdown prompt file (like the existing skills) that defines the AI's behavior for that conversation. These live in `skills/` alongside the existing skill files.

**`skills/tip.md`** - Guide the user to articulate what they learned. Ask clarifying follow-ups if the description is vague. Tag the learning with relevant categories. Save as a journal entry and flag for community sharing (tips can be surfaced to other users in future).

**`skills/stuck.md`** - Understand where the user is in their AI journey (read profile). Brainstorm ideas drawing from department resources, curriculum, and general AI knowledge. Suggest YouTube tutorials, specific tools, or exercises. Be encouraging and practical.

**`skills/brainstorm.md`** - Structured brainstorming session modeled on the Claude Code brainstorming skill. Help the user identify an opportunity to improve something at Digital Science. Walk through problem definition, potential AI solutions, feasibility, and next steps.

**`skills/wrapup.md`** - End-of-day reflection. Ask how the day went, what they worked on, what they learned, whether the program is helpful. Capture structured progress data. Generate a journal entry summarizing the day.

**`skills/intake.md`** - Placeholder. This is a deep design exercise covered separately. The intake prompt will handle first-run onboarding, proficiency assessment, role capture, and first-day recommendations.

## 3. Text Chat over WebSockets

### Why migrate from SSE

SSE (Server-Sent Events) is unidirectional: the server streams tokens to the client, but the client can only send messages via separate HTTP requests. Moving to WebSockets via API Gateway enables:

* **Server-initiated push** - background tasks (memory extraction, async tool calls) can notify the client without polling
* **Unified protocol** - same WebSocket connection for chat messages, status updates, and notifications
* **Connection decoupled from compute** - API Gateway holds the socket; Lambda is stateless and scales on CPU, not connection count
* **Deployment without disconnection** - deploying new Lambda code doesn't drop active user connections

### Architecture

```
                          WebSocket (persistent)
React Client  <=========================================> API Gateway
                                                          (WebSocket API)
                                                              |
                                    $connect / $disconnect / $default
                                                              |
                                                              v
                                                          Lambda
                                                     (stateless handler)
                                                              |
                                        +--------+--------+--------+
                                        v        v        v        v
                                    DynamoDB  S3      LanceDB  Bedrock
                                                              (Claude)
                                                                  |
                                                          tokens stream
                                                                  |
                                                                  v
                                                          Lambda pushes via
                                                          API GW Management API
                                                                  |
                                                                  v
                                                          API Gateway
                                                                  |
                                                                  v
                                                          React Client
```

### WebSocket authentication

API Gateway WebSocket APIs don't support Authorization headers on the initial handshake. Auth is handled via a Lambda authorizer on the `$connect` route:

1. Client includes the OIDC ID token as a query string parameter: `wss://{endpoint}?token={id_token}`
2. The `$connect` Lambda authorizer validates the JWT against ds-identity's JWKS endpoint (same logic as the current REST auth)
3. If valid, the connection is established; if not, the connection is rejected with 401
4. The token is only sent once (on connect); subsequent messages on the established connection are authenticated to the user_id from the token

The query string token is acceptable here because: (a) the connection is over WSS (TLS), (b) the token is short-lived, and (c) API Gateway does not log query strings by default.

**Authorization:** Authentication establishes identity, but each message must also be authorized. The Dispatcher/Worker validates that the `session_id` in every message belongs to the authenticated `user_id` by checking session ownership in DynamoDB. This prevents users from accessing other users' sessions via crafted messages.

### How it works

1. **Client connects** - React opens a WebSocket to the API Gateway endpoint with the auth token. API Gateway validates via the Lambda authorizer, assigns a `ConnectionId`, and invokes the `$connect` route handler. Lambda stores the ConnectionId + user_id mapping in DynamoDB.

2. **Client sends a message** - API Gateway invokes the `$default` Lambda route with the message payload and ConnectionId. Lambda runs the agent loop.

3. **Lambda streams response** - As the agent generates tokens, Lambda pushes them back to the client via the API Gateway Management API (`POST https://{api-id}.execute-api.{region}.amazonaws.com/{stage}/@connections/{connectionId}`). Each token/chunk is a separate Management API call.

4. **Background notifications** - Any Lambda (or other service) can push messages to a connected user by looking up their ConnectionId in DynamoDB and calling the Management API.

5. **Client disconnects** - API Gateway invokes the `$disconnect` route. Lambda removes the ConnectionId mapping.

### Connection state (DynamoDB)

```json
{
  "connection_id": "abc123=",
  "user_id": "oidc-sub-uuid",
  "connected_at": "2026-03-20T10:00:00Z"
}
```

A simple table with TTL-based cleanup for stale connections. The `$disconnect` handler deletes the record; TTL catches cases where disconnect wasn't clean.

**GSI for reverse lookup:** A `user_id-connected_at-index` GSI (PK=user_id, SK=connected_at) enables looking up all active connections for a given user. This supports background push notifications and handles multi-tab/multi-device scenarios where a user has multiple active ConnectionIds.

### API Gateway limits and mitigations

| Limit | Value | Mitigation |
|-------|-------|-----------|
| **Idle timeout** | 10 minutes | Client sends heartbeat ping every 5 minutes |
| **Connection duration** | 2 hours | Client auto-reconnects when connection closes |
| **Integration timeout** | 29 seconds per invocation | See below. |

### Handling the 29-second integration timeout

The API Gateway WebSocket integration timeout means the Lambda invoked by the `$default` route must return within 29 seconds. Since the agent loop (LLM calls, tool execution) can easily exceed this, we use a **Dispatcher-Worker pattern**:

1. **Dispatcher Lambda** - invoked by API Gateway's `$default` route. Validates the message, invokes the Worker Lambda asynchronously (`InvocationType='Event'`), and returns 200 immediately. Runs in <1 second.

2. **Worker Lambda** - runs the agent loop (up to Lambda's 15-minute timeout). Pushes tokens, tool calls, and status messages back to the client via the API Gateway Management API. Checks a cancel flag in DynamoDB periodically so it can abort if the client disconnects or cancels.

```
Client → API GW → Dispatcher Lambda (returns 200 immediately)
                        ↓ async invoke
                   Worker Lambda (runs agent loop, pushes tokens via Management API)
                        ↓
                   API GW → Client
```

This keeps the architecture fully serverless while avoiding the timeout constraint. The Dispatcher is lightweight and cheap; the Worker is where the compute happens.

**Session-level mutex:** The Dispatcher sets a `processing` flag in DynamoDB (conditional put) before invoking the Worker. If the flag is already set, the Dispatcher returns a `{"type": "error", "message": "Session is already processing"}` to the client instead of spawning a second Worker. The Worker clears the flag on completion. A TTL-based expiry (e.g., 5 minutes) ensures crashed Workers don't permanently lock a session.

**Error handling:** If the async Worker invocation fails (Lambda throttled, service error), the Dispatcher catches the error and pushes `{"type": "error", "session_id": "...", "message": "..."}` to the client via the Management API. The user should never send a message and get silence.

**Token batching:** The Worker accumulates tokens for 50-100ms before sending a single Management API call with the batch. This reduces API calls by ~10-20x compared to per-token sends, lowering latency from network round-trips without perceptible delay to the user.

**Frame size limits:** API Gateway WebSocket has a 128 KB server-to-client frame limit and 32 KB client-to-server limit. The Worker checks message size before sending via the Management API. Messages exceeding the limit are chunked into multiple frames with sequence numbers; the client reassembles them. Search results are truncated to fit within limits.

### Message protocol

Client-to-server (via WebSocket):
```json
{"action": "start_session", "type": "tip", "mode": "text"}
{"action": "chat", "session_id": "uuid", "message": "How can I use AI for..."}
{"action": "cancel", "session_id": "uuid"}
{"action": "ping"}
```

Server-to-client (via Management API):
```json
{"type": "token", "session_id": "uuid", "content": "Here's"}
{"type": "token", "session_id": "uuid", "content": " how"}
{"type": "tool_call", "session_id": "uuid", "tool": "search", "args": {...}}
{"type": "tool_result", "session_id": "uuid", "tool": "search", "result": {...}}
{"type": "done", "session_id": "uuid"}
{"type": "error", "session_id": "uuid", "message": "..."}
{"type": "pong"}
```

### CDK changes

Add to the CDK stack:
* API Gateway WebSocket API with `$connect`, `$disconnect`, and `$default` routes
* Lambda integrations for each route (can be same Lambda with route-based dispatch, or separate)
* DynamoDB connections table
* IAM permissions for Lambda to call the API Gateway Management API
* CloudFront behavior routing `/ws` to the WebSocket API endpoint

The existing Lambda Function URL and SSE streaming can be removed once WebSocket migration is complete.

### Frontend changes

Replace the SSE fetch-based streaming with a WebSocket client:
* Single persistent connection opened at app startup (after auth)
* Auto-reconnect with exponential backoff on disconnect
* Heartbeat ping every 5 minutes
* Message dispatch based on `type` field
* Same React state management (useReducer), just a different transport
* **Offline message queue:** Messages typed while disconnected are queued locally and sent automatically when the connection restores. A thin "Reconnecting..." banner appears below the header during disconnection (see Section 8 for full spec)

## 4. Voice Mode (OpenAI Realtime API)

### Why OpenAI Realtime, not Nova Sonic

Amazon Nova Sonic on Bedrock requires the client to send the system prompt and tool definitions in the initial stream configuration. This means any user could see the prompts in browser dev tools. OpenAI's Realtime API supports server-side session creation with ephemeral tokens: the backend configures the session (system prompt, tools, context), and the client receives an opaque token that grants access to that pre-configured session without ever seeing the prompt.

### Architecture: Client-Direct Ephemeral Token

Voice audio never touches our AWS infrastructure. The backend's role is limited to session creation and tool execution.

```
                     1. {"action": "voice_session", "session_id": "..."}
React Client  =======================================> Lambda (via API GW WebSocket)
      |        <======================================= {"type": "voice_token", "token": "eph_..."}
      |
      |              2. Bidirectional audio stream
      +---------------------------------------------> OpenAI Realtime API (GPT-4o)
      |        <-------------------------------------  wss://api.openai.com/v1/realtime
      |
      |              3. Tool calls (when needed)
      +--------- tool_call event from GPT-4o -------> Client intercepts
      |         {"action": "tool_call", ...}
      +---------------------------------------------> Lambda (via API GW WebSocket)
      |        <======================================= {"type": "tool_result", ...}
      +--------- tool_result pushed to GPT-4o ------> GPT-4o continues
```

### Session creation

Client requests a voice session via the existing WebSocket connection:

```json
{"action": "voice_session", "session_id": "uuid", "type": "intake"}
```

Lambda handles this by:

1. Loading the user's profile and the session-type prompt (e.g., intake, brainstorm)
2. Calling the OpenAI Realtime session endpoint with the system prompt, tool definitions, and user context
3. Receiving an ephemeral token scoped to that session
4. Pushing the token back to the client via the Management API

```json
{"type": "voice_token", "session_id": "uuid", "token": "eph_abc123", "expires_at": "2026-03-20T11:30:00Z"}
```

The system prompt and tool definitions are configured server-side. The client never sees them. The ephemeral token is short-lived and scoped to a single session.

### Tool execution during voice

When GPT-4o decides to call a tool, it sends a tool_call event down the audio WebSocket to the browser. The React client:

1. Intercepts the tool_call JSON event
2. Extracts the tool name and parameters
3. Sends the tool call to the backend via the API Gateway WebSocket (`{"action": "tool_call", "tool": "search", "args": {...}}`)
4. Lambda validates the tool call (see below), executes it, and pushes the result back via the Management API
5. Client receives the tool result and pushes it to GPT-4o via the voice WebSocket
6. GPT-4o continues the voice response incorporating the tool result

This keeps tool execution on Lambda (serverless, secure, has access to DynamoDB/S3/LanceDB) while audio stays client-direct. The tool call path reuses the same WebSocket connection used for text chat.

**Server-side tool validation:** Since the client relays tool calls from GPT-4o, the browser is an untrusted intermediary. Lambda validates every tool call before execution: (1) tool name must be in the allowlist of registered tools, (2) arguments must pass schema validation, (3) the ephemeral token is bound to a session_id server-side so Lambda can verify the tool call originates from a legitimate voice session. Fabricated or tampered tool calls are rejected with an error.

### Frontend voice UI

* **Mode toggle:** A microphone icon on the chat input bar. Click to switch between text and voice mode. Voice activates immediately; the model's greeting serves as implicit orientation (no tooltip).
* **Voice active state:** Replaces the text input with a warm pulsing orb (~120px diameter, brand blue `#159AC9` with subtle gradient). The orb has distinct visual states: gentle breathing pulse while listening (expands on user voice amplitude), rhythmic expansion while AI speaks, subtle slow pulse when idle, and a different "searching" pattern during tool calls. See Section 8 for full spec. Shows a "Stop" button.
* **Transcript:** Voice interactions are transcribed in real-time (GPT-4o provides transcription events for both user and assistant speech) and displayed as chat messages in the same session, scrolling below the voice orb.
* **Mid-conversation switching:** Users can start in voice and switch to text (or vice versa) at any point. The session and context are preserved via the shared S3 transcript.
* **Session type compatibility:** Voice works with all session types. User can click "Share a Tip" and choose to do it via voice.
* **Microphone permission denied:** If the browser denies mic access, show a friendly inline message with browser-specific instructions to re-enable, plus a "Try Again" button. Don't hide the voice toggle.

**Mode switching behavior:**
* **Voice → Text:** Clicking the text toggle ends the OpenAI Realtime WebSocket connection (any in-progress voice response is interrupted). The transcript up to that point is already on S3. The next message typed by the user goes through the normal Claude/Bedrock agent loop, which reads the full transcript (including voice turns) from S3. Claude sees a system note: "Prior messages may include voice conversation turns from a different model."
* **Text → Voice:** Clicking the voice toggle requests a new ephemeral token via `{"action": "voice_session", ...}`. Lambda includes the existing transcript as context in the Realtime session's system prompt. GPT-4o continues the conversation aware of prior text turns. Any in-progress text stream is allowed to finish before the voice connection opens.
* **Back-and-forth:** Each voice toggle creates a new Realtime session (ephemeral tokens are single-use). This is fine - the transcript provides continuity. There is no limit on how many times a user can switch.

### Voice transcript persistence

Voice conversations are persisted as text transcripts, identical to text chat sessions. GPT-4o Realtime provides transcript events for both user speech and model speech.

The React client captures these transcript events and sends them to the backend via the API Gateway WebSocket (`{"action": "transcript", "session_id": "...", "role": "user"|"assistant", "content": "..."}`). Lambda appends them to the S3 session transcript. Voice sessions have a full text record even though the audio itself is not stored.

### Voice session recovery

With 700 users doing 10-15 minute voice conversations, disconnects will happen (WiFi drops, tab refresh, laptop sleep). The architecture supports recovery because transcript chunks are relayed to Lambda in near-real-time and persisted to S3 as the conversation progresses.

**Recovery flow:**

1. Client detects the OpenAI Realtime WebSocket has closed (onclose event)
2. Client displays a "Reconnecting..." indicator and auto-reconnects to the API Gateway WebSocket (if also dropped)
3. Client sends `{"action": "voice_session", "session_id": "uuid", "resume": true}` to request a new ephemeral token for the same session
4. Lambda reads the existing transcript from S3, includes it as conversation context in the new OpenAI Realtime session's system prompt ("You are resuming a conversation. Here is the transcript so far: ...")
5. GPT-4o picks up the conversation naturally ("I see we were just discussing... Let's continue.")
6. New transcript events are appended to the existing S3 transcript

**What's preserved:** Full text transcript up to the last relayed chunk (typically within a few seconds of the disconnect). Profile data, proficiency scores captured so far, and tool results.

**What's lost:** The last few seconds of unrelayed transcript, and GPT-4o's voice personality/inflection calibration (resets with new session). These are acceptable tradeoffs.

**Intake-specific handling:** The intake prompt (`skills/intake.md`) includes a resume instruction: if the transcript already contains conversation history, skip completed phases and continue from where the conversation left off. The AI should acknowledge the interruption briefly ("Welcome back! We were talking about...") rather than starting over.

### OpenAI API key management

The OpenAI API key is stored in AWS SSM Parameter Store (same pattern as ds-internal-mcp credentials). Lambda reads it at invocation time to create ephemeral sessions. The key never reaches the client.

## 5. Search and RAG

### Extending existing search for shared tables

The current design has a curriculum-specific LanceDB table that was never indexed in production. Forge will connect to the same shared LanceDB tables that ds-internal-mcp uses.

### Shared search infrastructure

**S3 bucket:** `s3://forge-ds-databases/lance/`

Both Forge and ds-internal-mcp read from the same LanceDB tables:

| Table | Content | Use in Forge |
|-------|---------|-------------|
| `gong_turns` | Sales call transcript chunks | Understanding customer conversations, sales context |
| `gong_calls` | Call summaries | High-level call context |
| `dovetail_highlights` | User research quotes | Product/UX insights |
| `dovetail_notes` | Research session notes | Research context |
| `roadmap` | Product roadmap items | Understanding what teams are building |
| `klue_battlecards` | Competitive intelligence | Market context |
| `department_resources` | Admin-managed per-function content (NEW) | Department-specific guidance and links |

### Search pipeline

Forge's existing search pipeline (`backend/lance/`) already implements the same core algorithm as ds-internal-mcp:

1. **Embed query** - Cohere Embed v3 via Bedrock (1024-d, `input_type="search_query"`)
2. **Hybrid search** - FTS + vector similarity, combined via Reciprocal Rank Fusion (RRF)
3. **Rerank** - Cohere Rerank v3.5 via Bedrock (cross-encoder)
4. **Post-process** - Score gap detection (cut at >40% drop)

### Code approach: extend, don't copy

The existing `backend/lance/search.py`, `reranking.py`, `connection.py`, and `schemas.py` (~720 lines) are table-agnostic and need only two additions to support the shared tables:

1. **Federated search** - a wrapper that queries multiple tables per search call, deduplicates results via content hash, and merges ranked results. The existing single-table search becomes the inner loop.
2. **Name boosting** - parameterized per-schema (each schema definition includes a `name_fields` list for metadata-based score boosting at 1.25x per match). This is cleaner than ds-internal-mcp's hardcoded field list.

No code is copied from ds-internal-mcp. Both repos read the same LanceDB tables on S3 but maintain independent search code. The core algorithm is identical; any future divergence in ranking tuning is acceptable since the repos serve different use cases.

### Tools update

Replace the current `search_curriculum` tool with a more general `search` tool:

```python
def search(
    query: str,
    tables: list[str] | None = None,  # None = search all
    limit: int = 20,
    filter_expr: str | None = None,
) -> dict
```

The agent decides which tables to search based on context. For a marketing user asking about competitive positioning, it might search `klue_battlecards` and `department_resources`. For someone asking about product direction, it might search `roadmap` and `dovetail_highlights`.

The `retrieve_document` tool stays as-is for fetching full documents from S3.

## 6. Department Resources

### Purpose

Each department/function has admin-managed content that the AI uses when making personalized suggestions. This replaces the static curriculum files.

### Structure

Department resources are markdown files stored on S3. There are two kinds: a global resource file with cross-functional suggestions (e.g., optimizing your calendar, prompt engineering basics), and per-function files with department-specific guidance.

```
s3://{bucket}/department-resources/
  global.md              # Cross-functional suggestions for everyone
  technology.md
  marketing.md
  sales.md
  product.md
  legal.md
  finance.md
  people.md
  leadership.md
```

Each file contains:
* Getting started guidance (global or function-specific)
* Links to external resources (Google Drive docs, YouTube videos, tools)
* Function-specific AI use cases and suggestions
* Skill-level-appropriate recommendations (beginner through advanced)

Example structure for a department resource file:

```markdown
# Marketing - AI Tuesdays Resources

## Getting Started
Welcome to AI Tuesdays! Here are some ways marketing team members
can start using AI effectively...

## By Role

### Content Marketing
* [Content creation with Claude](https://drive.google.com/...)
* Try: Use AI to generate 5 variations of a blog post headline
* Advanced: Build a content calendar assistant

### Social Media
* [Social media AI toolkit](https://drive.google.com/...)
* Try: Analyze engagement data from last month with AI
...

## Recommended Tools
* ChatGPT for brainstorming
* Claude for long-form writing
* Midjourney for visual assets
...
```

### Indexing

Department resources are indexed into a `department_resources` LanceDB table using the same chunking and embedding pipeline as other content. Metadata includes `department` for filtering.

### Admin management

**Phase 1 (now):** Department leads edit markdown files directly. Upload to S3 via CLI or a simple upload script. Re-index after changes.

**Phase 2 (future):** Admin UI in Forge where department leads can edit their function's resources in a markdown editor with preview. Changes auto-index.

## 7. Intake

The intake is the first-run experience for every new user. It blocks access to everything else until complete. It serves dual purposes: personalize the user's AI Tuesday experience, and build organizational intelligence about people, skills, and teams that Digital Science currently lacks.

### Design principles

* **Feels like a coach, not an interview.** The AI is genuinely curious, gives insights along the way, and leaves the user excited about what's possible. Not an HR form.
* **Give before you take.** After every 2-3 questions, offer a micro-insight or observation. ("That's actually a really common pattern in marketing teams, and there's a great AI approach for it.")
* **Never grade.** The AI never says "you're a beginner." It says "you've got a solid foundation with ChatGPT - there are some techniques that could take that further."
* **Spark ideas.** Don't just capture what people already know. Help them discover possibilities. ("You mentioned you spend time on WordPress updates - have you tried using AI to draft content and optimize SEO simultaneously?")
* **Voice-friendly.** All questions are open-ended enough for someone to talk for 30 seconds. No yes/no questions.
* **Social proof.** Normalize participation: "A lot of people in similar roles find that..."

### Conversation flow (~10-15 minutes)

**Phase 1: Warm-up and validation (~1 min)**
Confirm org chart data (name, title, department, manager). Fix anything wrong. Low-friction start to build rapport.

**Phase 2: What you do (~3 min)**
* "Tell me about what you actually work on day to day. What products are you involved with?"
* "Walk me through a typical week - what takes up most of your time?"
* "What's the stuff that feels like busy work - things you wish you could hand off?"
* Captures: products, daily tasks, pain points, delegation opportunities

**Phase 3: Skills and background (~2 min)**
* "What would you say your core skills are? The things you're genuinely good at?"
* "Is there anything you'd love to learn or get better at if you had the time?"
* "Have you worked on anything recently that you're particularly proud of?"
* Captures: self-assessed skills, aspirations, recent wins

**Phase 4: AI experience - deep behavioral assessment (~3 min)**
Not self-reported skill levels. Behavioral questions that the AI maps internally to proficiency dimensions:
* "Have you used any AI tools yet? Which ones, and what for?" (operational fluency)
* "What's the most useful thing you've done with AI so far?" (strategic delegation)
* "Have you ever had AI give you something wrong or off? How did you catch it?" (discernment)
* "When you're thinking about using AI for something at work, how do you decide what's safe to share?" (security awareness)
* "Do you have any recurring tasks where you find yourself prompting AI the same way every time?" (automation readiness)

**Phase 5: Aspirations and superpowers (~2 min)**
* "If AI could give you one superpower at work, what would it be?"
* "Is there something you see other teams or companies doing with AI that you wish you could do?"
* "What would success look like for you after these 12 weeks?"
* Captures: goals, vision, motivation

**Phase 6: Personalized first-day suggestions (~2 min)**
AI synthesizes everything, reads the user's department resource file (e.g., marketing.md) plus global.md, and generates 3-4 tailored suggestions for their first AI Tuesday. Each suggestion maps to their role, skill level, and stated interests. Ends with encouragement and orientation to the action buttons.

### AI proficiency dimensions (scored internally)

The AI infers scores from the behavioral conversation. Users never see numbers. Five dimensions based on AI readiness research:

| Dimension | What it measures | Inferred from |
|-----------|-----------------|---------------|
| **Operational fluency** | Proficiency with AI tools, prompting techniques | Which tools they use, what they've built |
| **Strategic delegation** | Ability to identify AI-appropriate tasks | How they describe their workflow, what they'd hand off |
| **Discernment** | Detecting errors, hallucinations, bias in AI output | Whether they've caught AI mistakes, how they verify |
| **Security awareness** | Data privacy instincts, appropriate use | How they decide what to share with AI tools |
| **Automation readiness** | Moving from chatting to building workflows | Whether they have recurring prompts, interest in integration |

Each scored 1-5 by the AI. Stored in the profile for matching, progress tracking, and organizational analytics.

### Two-model pattern for voice intake

In voice mode, the real-time voice model (GPT-4o) handles conversation. But the heavy analytical work - proficiency scoring, personalized insights, first-day suggestions - routes to Claude Opus via a tool call.

```
User  <--voice-->  GPT-4o (conversational, real-time)
                      |
                      | calls "analyze_and_advise" tool
                      v
                   Client relays tool call (session_id + question only,
                   no transcript passes through browser)
                      |
                      v
                   Lambda reads full transcript from S3
                   → calls Claude Opus via Bedrock
                   → returns insights/scores/recommendations
                      |
                      v
                   Client relays result back to GPT-4o
                      |
                      v
                   GPT-4o speaks the Opus-generated insights
```

This ensures:
* Conversational feel stays fast and natural (GPT-4o)
* Insights are genuinely insightful (Opus does the thinking)
* User experiences one seamless conversation
* The tool call payload only carries session_id and question - the full transcript is already on S3 (persisted via the transcript relay described in Section 4) and Lambda reads it directly
* For text mode (already on Claude via Bedrock), this pattern isn't needed

The `analyze_and_advise` tool is called at key moments: after phases 2-3 (to generate micro-insights), after phase 4 (to score proficiency), and at phase 6 (to generate first-day suggestions). Transcript must be persisted in near-real-time during voice sessions so it's available when Opus is called mid-conversation.

### New tool: analyze_and_advise

Used during voice intake to route analytical work to Claude Opus:

```python
def analyze_and_advise(
    session_id: str,
    question: str,  # e.g., "score proficiency dimensions" or "generate first-day suggestions"
) -> str
```

Lambda reads the session transcript from S3, sends it to Opus with the question, returns the response. The voice model (GPT-4o) speaks the result. In text mode, this tool is unnecessary since the agent already runs on Claude.

### Blocking behavior

The intake blocks access to all other features until complete:
* Frontend checks `intake_completed_at` on the profile
* If null, the only available action is the intake session
* **Full-screen focused layout:** No sidebar, no action buttons, no session history. Centered chat area with a warm welcome header ("Welcome to AI Tuesdays / Let's get to know each other.") and a 6-dot phase progress indicator (filled = completed phases). Voice toggle available from the start. "Need help?" link pinned at the bottom. See Section 8 for full spec.
* The intake session is auto-created on first sign-in, tagged as type `intake`

**Resilience for a 700-user rollout:**
* **Auto-save:** Profile fields are saved incrementally as the AI captures them (after each phase), not just at completion. If the session crashes at phase 4, phases 1-3 data is preserved.
* **Resume:** If a user returns to an incomplete intake, the existing intake session is loaded (not recreated). The transcript is on S3; the AI reads it and continues from where it left off. For voice, the resume flow in Section 4 applies.
* **Clear error messages:** If the voice connection fails or the AI encounters an error, show a clear message with a "Try again" button. "Something went wrong. Your progress is saved - click to continue where you left off."
* **Support contact:** A "Need help?" link is always visible during intake, connecting to the program administrator.

### Screen States and Hierarchy

**Intake (first-run, blocks everything):**
Full-screen focused layout. No sidebar, no action buttons, no session history. Centered chat area with warm welcome header and a subtle 6-dot phase progress indicator (filled dots = completed phases). "Need help?" link pinned at the bottom. Voice toggle available in the input bar from the start so users can do intake via voice.

**Home (intake complete, no active session):**
Context-aware greeting that varies by day and activity:
* Tuesday: "Ready for AI Tuesday, [Name]?"
* After first session: "Nice tip last week! What's next?"
* After 5+ sessions: "Week [N] of 12. You're building momentum."
* Default: "Welcome back, [Name]."

Four action buttons in a 2x2 grid below the greeting. Standard sidebar with session history visible.

**Active session (text mode):**
Same as current layout. Message thread is primary, input bar with voice toggle is secondary, sidebar with session list is tertiary.

**Active session (voice mode):**
Primary: Warm pulsing orb (~120px diameter, centered) using `--color-active-blue` with a subtle gradient. States:
* **Listening:** Gentle breathing pulse, expands slightly on user voice amplitude
* **AI speaking:** Rhythmic expansion, brighter blue pulses with speech cadence
* **Idle:** Subtle slow pulse, muted
* **Tool call:** Different pattern (searching indicator), audio cue from model ("Let me check that...")

Secondary: Live transcript scrolling below the orb. Tertiary: Stop button + text toggle, sidebar.

### Session List Design

Sessions in the sidebar are grouped by week ("This Week", "Last Week", "Week 3", etc.). Current week expanded by default, older weeks collapsed. A search bar at the top filters across all sessions.

Each session entry shows a type icon + title + relative time:
* Tip: lightbulb icon
* Stuck: compass icon
* Brainstorm: star icon
* Wrap-up: sunset icon
* Chat: speech bubble icon
* Intake: (not shown in list while incomplete)

No icon legend in the UI; icons are self-explanatory.

### Interaction States

**Action button click:** Instant transition to new session view with existing streaming dots indicator. Session appears in sidebar simultaneously. No intermediate loading screen.

**WebSocket disconnect (text mode):** Thin banner below header: "Reconnecting..." with subtle pulsing animation. Input stays enabled; typed messages are queued and sent automatically when connection restores. Banner auto-dismisses on reconnect.

**Voice reconnection:** Semi-transparent overlay on the voice visualizer showing "Reconnecting... Your progress is saved." Transcript remains visible underneath so the user can see where they left off. Auto-dismisses when voice resumes.

**Voice tool call gap:** The voice model says a brief filler ("Let me check that for you...") before invoking tools. The orb switches to a distinct "searching" animation pattern. Visual + audio cues together prevent the "did it freeze?" anxiety.

**Microphone permission denied:** If the browser mic permission is denied, show a friendly inline message explaining how to enable the microphone in browser settings, with a "Try Again" button. Don't hide the voice toggle; show the explanation when tapped. This handles accidental denials.

**Voice activation (first or subsequent use):** Voice mode activates immediately on tap. No orientation tooltip. The voice model's greeting ("How can I help you today?") serves as implicit orientation.

### Responsive Design

**Mobile text mode:** Existing pattern (sidebar hidden behind hamburger, full-width chat).

**Mobile voice mode (375px):**
* Orb centered in the top third of the screen, full width
* Live transcript scrolling below the orb (slightly smaller text)
* Bottom bar: text-toggle button [Aa] on left, stop button on right
* Sidebar hidden (standard mobile pattern)

**Tablet (768px-1024px):** Same as desktop but with collapsible sidebar.

### Accessibility

* **Keyboard navigation:** Tab through action buttons in reading order, Enter/Space to activate. Voice toggle reachable via Tab from the input bar.
* **ARIA live region:** Voice status changes ("Listening", "AI is speaking", "Reconnecting") announced via `aria-live="polite"` region so screen reader users understand mode transitions.
* **Touch targets:** All interactive elements are 44px minimum on mobile (action buttons, voice toggle, stop button, sidebar items).
* **Focus management:** When transitioning to intake or a new session, focus moves to the first interactive element (input bar or voice toggle).

## 10. API Changes

### New: WebSocket API (replaces SSE chat endpoint)

```
WebSocket routes (API Gateway):
  $connect                          Auth + store ConnectionId
  $disconnect                       Remove ConnectionId
  $default                          Route by action field

Client-to-server actions:
  {"action": "start_session", ...}  Create typed session + initiate conversation
  {"action": "chat", ...}           Send a chat message
  {"action": "cancel", ...}         Cancel a running stream
  {"action": "voice_session", ...}  Request ephemeral voice token (or resume)
  {"action": "tool_call", ...}      Execute a tool (voice mode, server-validated)
  {"action": "transcript", ...}     Persist voice transcript chunk
  {"action": "ping"}                Heartbeat

Server-to-client messages:
  {"type": "token", ...}            Streaming token
  {"type": "voice_token", ...}      Ephemeral OpenAI token for voice session
  {"type": "tool_call", ...}        Tool call notification
  {"type": "tool_result", ...}      Tool result
  {"type": "done", ...}             Stream complete
  {"type": "error", ...}            Error
  {"type": "pong"}                  Heartbeat response
```

### Modified endpoints

```
POST   /api/sessions                Now accepts optional {type: "tip"|"stuck"|...}
GET    /api/sessions                Now returns type field; supports ?type= filter
```

### Removed

```
POST   /api/chat                    Replaced by WebSocket {"action": "chat"}
POST   /api/chat/cancel             Replaced by WebSocket {"action": "cancel"}
```

### Unchanged

All other REST endpoints (sessions CRUD, profile, journal, ideas, health) remain as-is.
