# Intake Completion Flow - Implementation Spec

## Background

Forge is an AI coaching companion for Digital Science's "AI Tuesdays" program. When a
new user opens the app for the first time, they go through an "intake" flow:

1. **Onboarding cards** (`frontend/src/components/OnboardingCards.tsx`) - 4 swipeable
   full-screen cards explaining what AI Tuesdays is
2. **Intake conversation** (`frontend/src/components/IntakeView.tsx`) - An AI-driven
   conversation that learns about the user's role, work, AI experience, and goals
3. **Main app** (`frontend/src/App.tsx` renders `AppShell` with sidebar + home screen) -
   Where they access session types: Tip, Stuck, Brainstorm, Wrap-up

The transition from step 2 to step 3 is currently broken. When the intake finishes,
the page silently refreshes, the sidebar appears, and the user has no idea what just
happened or what to do next.

## Architecture context

### How the app decides what to show

`frontend/src/App.tsx` (`AppContent` component):
- Fetches the user profile via `getProfile()` REST call
- Checks `profile.intake_completed_at`
- If null: renders `<IntakeView />` (full screen, no sidebar)
- If set: renders `<AppShell sidebar={sidebar} content={content} />` (sidebar + home/chat)

### How intake completion is currently detected

`backend/agent/executor.py` (`_check_intake_completion` function):
- Runs after every agent turn in an intake session
- Checks how many user messages exist and how many profile fields were captured
- When threshold is met, calls `profiles_repo.update()` to set `intake_completed_at`
- App.tsx re-fetches the profile after each streaming response ends (line ~42-48)
- When it sees `intake_completed_at` is now set, React re-renders and swaps IntakeView for AppShell

### How sessions work

- Sessions are created via WebSocket: `{action: "start_session", type: "intake"}`
- Backend creates a `Session` record in DynamoDB/memory with `session_id`, `user_id`, `title`, `type`
- Titles are auto-generated by the LLM after the first exchange
- Sessions appear in the sidebar (`frontend/src/components/SessionList.tsx`)
- Sessions can be deleted via `deleteSession()` REST call

### Key files

| File | Role |
|------|------|
| `frontend/src/App.tsx` | Routes between IntakeView and AppShell based on profile state |
| `frontend/src/components/IntakeView.tsx` | Full intake experience (cards + chat) |
| `frontend/src/components/ChatView.tsx` | Normal chat view (inside AppShell) |

| `backend/lambda_ws.py` | WebSocket handler (production Lambda) |
| `backend/api/sessions.py` | REST endpoints for session CRUD |

## What needs to change

### Problem 1: No completion ceremony

**Current:** When intake completes, `intake_completed_at` gets set on the profile.
App.tsx detects this on re-render and swaps IntakeView for AppShell. The user sees
the page flicker and suddenly they're in a different UI with a sidebar. No explanation.

**Fix:** When the backend determines intake is complete, send a new WebSocket message
`intake_complete` to the frontend. IntakeView renders a **completion card** (a styled
UI element, not an AI message) at the bottom of the conversation:

Clicking the button transitions to the main app (home screen with action buttons).

**Implementation:**

Backend (`executor.py`): In `_check_intake_completion`, when completion triggers,
send a WebSocket message before the `done` message:
```python
await sender.send({
    "type": "intake_complete",
    "session_id": session_id,
})
```

Frontend (`websocket.ts`): Add `intake_complete` to the `ServerMessage` union type.

Frontend (`SessionContext.tsx`): Handle `intake_complete` message:
```typescript
case 'intake_complete':
  dispatch({ type: 'SET_INTAKE_COMPLETE', sessionId: msg.session_id });
  break;
```

Frontend (`IntakeView.tsx`): When `state.intakeComplete` is true, render the
completion card below the messages. The "Continue" button calls:
```typescript
function handleContinue() {
  // Re-fetch profile so App.tsx sees intake_completed_at
  getProfile().then(p => setProfile(p));
}
```

This triggers App.tsx to swap IntakeView for AppShell, showing the home screen.

### Problem 2: Intake session has no proper name

**Current:** The session title is auto-generated by the LLM after the first exchange.
It gets names like "New Conversation Introduction" or "AI Tuesdays Welcome Meeting".

**Fix:** Hardcode the intake session title to "Getting Started".

Backend (`websocket.py` line ~138, `_handle_start_session`): When `session_type == "intake"`,
set `session.title = "Getting Started"` before creating the session. Skip the LLM
title generation for intake sessions.

Backend (`lambda_ws.py` line ~271, `_worker_start_session`): Same change.

Backend (`executor.py` line ~298): Skip the `_generate_title` call when
`session_type == "intake"`.

### Problem 3: Intake session shouldn't be deletable

**Current:** All sessions can be deleted from the sidebar context menu.

**Fix:** Hide the delete option for intake sessions (type == "intake") unless the
user is an admin or in dev_mode.

Frontend (`SessionList.tsx`): In the session context menu rendering, check
`session.type !== 'intake'` before showing the delete button. In dev_mode
(check `window.location.hostname === 'localhost'`), always show delete.

Backend (`api/sessions.py`): In the delete endpoint, reject deletion of intake
sessions unless the requesting user is an admin. For now, check `settings.dev_mode`
as the admin proxy.

### Problem 4: Should navigate to home screen after completion

**Current:** After the profile refresh triggers the AppShell, the user lands on
whatever session was active (the intake session), showing the conversation with
all the tool call artifacts.

**Fix:** When transitioning from IntakeView to AppShell after the completion card:
1. Call `deselectSession()` from SessionContext (sets `activeSessionId` to null)
2. This causes AppShell to render `HomeScreen` (the action buttons view)
3. "Getting Started" appears in the sidebar session list for future reference

The `handleContinue` function in IntakeView should:
```typescript
function handleContinue() {
  deselectSession();
  // Small delay then re-fetch profile to trigger the view swap
  setTimeout(() => {
    getProfile().then(p => setProfile(p));
  }, 100);
}
```

## Testing checklist

- [ ] Fresh user: sees cards, then chat, then completion card, then home screen
- [ ] Returning user (intake in progress): skips cards, sees chat
- [ ] Returning user (intake complete): goes straight to AppShell
- [ ] "Getting Started" appears in sidebar after completion
- [ ] "Getting Started" cannot be deleted (except in dev mode)
- [ ] Completion card shows personalized suggestions
- [ ] "Continue" button navigates to home screen, not the intake chat
- [ ] Tool calls not visible in the session transcript (except dev mode)
